# FORGE Constitution — 05: AI Ethics and Governance

These rules govern how AI is used responsibly within the FORGE framework. They complement the technical rules in files 01-04 and apply to every AI session, regardless of tool or stage.

---

## 1. Transparency: AI Work Must Be Identifiable

**Rule:** Any artifact produced with significant AI assistance must be labeled as such.

**What this means in practice:**
- Generated files include a header: `Generated by: [AI tool]  Date: [date]`
- Generated sections in human-authored documents are marked: `[AI-assisted section — reviewed by: [name]]`
- Commit messages reference AI assistance when code was primarily generated: `Co-Authored-By: [tool]`
- PRs disclose AI-generated files in the description

**Why:** Reviewers need to know whether they are reviewing human judgment or AI output. The review rigor differs. Hiding AI provenance undermines the governance model.

---

## 2. Human Authority: AI Advises, Humans Decide

**Rule:** AI may produce options, analysis, recommendations, and drafts. Humans make every consequential decision.

**Consequential decisions include:**
- Architecture choices (service boundaries, data ownership, technology selection)
- Business rule interpretation (what a legacy system was intended to do)
- Security risk acceptance (accept a finding vs fix it)
- Production deployment (go / no-go)
- Regulatory compliance statements
- Any decision that affects customer data or financial outcomes

**What this means in practice:**
- Review gates exist precisely because AI output is not ground truth
- An AI-generated architecture is a proposal, not a decision
- An AI-generated business rule is a hypothesis, not a specification
- "The AI said so" is never sufficient justification for a production change

---

## 3. Bias Awareness: Verify AI Recommendations Against Actual Requirements

**Rule:** AI models are trained on general patterns. Verify that recommendations fit the specific context.

**Common AI biases to watch for:**
- **Recency bias:** AI may favor the latest popular frameworks over the right tool for the job
- **Greenfield bias:** AI trained on examples tends to propose clean-slate designs; brownfield systems have constraints that must be respected
- **Technology bias:** AI may suggest a tool it knows well over a better-fit alternative it knows less well
- **Over-engineering:** AI tends to propose complete, elaborate solutions even when a simpler one is sufficient
- **False confidence:** AI presents uncertain analysis with the same tone as certain analysis; confidence scores exist for this reason

**What this means in practice:**
- When AI recommends a technology, ask: "Does this fit our team's skills, our existing platform, our NFRs?"
- When AI extracts a business rule, ask the SME: "Is this actually how it works?"
- When AI proposes an architecture, ask the Lead Engineer: "Can we actually build and operate this?"

---

## 4. Data Responsibility: What Must Not Enter an AI Prompt

Detailed rules are in `constitution/02-security-and-data.md`. Summary:

| Data Type | Rule |
|---|---|
| Production PII | Never — use synthetic or anonymized data |
| Credentials / API keys | Never |
| PAN (card numbers) | Never — mask to last 4 |
| Production database connection strings | Never |
| Customer names, SSNs, account numbers | Never — use TEST- prefixed values |
| Internal IP addresses / hostnames | Review — only if already public |
| Proprietary business logic | Assess — is it competitive IP? |

**Responsibility chain:**
1. Every team member is responsible for reviewing prompts before submitting
2. Tech Lead is responsible for confirming pre-prompt safety checklist (in `guardrails/README.md`) is completed before any AI session involving source code
3. Security Engineer reviews AI session patterns during security gates

## 5. Intellectual Property: AI Output and Ownership

**Rule:** Understand the IP implications of AI-generated code before committing it.

**What to verify before using AI-generated code:**
- Your AI tool's terms of service address code ownership (most enterprise tools transfer ownership to the user)
- AI-generated code that closely resembles open-source training data may carry license obligations
- For regulated industries: confirm your organization's AI code generation policy

**FORGE guidance:**
- Use enterprise-licensed AI tools (GitHub Copilot Enterprise, watsonx.ai, Claude API with enterprise agreement) — not consumer tools for production code
- If using a consumer tool, have legal review any novel/complex generated code
- Document which AI tool produced which artifacts in the delivery package

---

## 6. Quality Assurance: AI Output is a First Draft

**Rule:** AI-generated artifacts must go through the same review process as human-produced artifacts — or a more rigorous one.

**The review bar does not lower because AI produced it. It may be higher because:**
- AI can be confidently wrong
- AI output may look correct but be subtly wrong (e.g., security vulnerability that is syntactically valid)
- AI doesn't know your production environment, your edge cases, your team's operational experience

**What this means in practice:**
- AI-generated code must pass the same code review checklist as human code (Stage 05 P2)
- AI-generated business rules must be validated by an SME — not just a developer
- AI-generated architecture must be reviewed by a qualified architect
- AI-generated test cases do not replace test planning — they are a starting point

---

## 7. Continuous Improvement: Learn From AI Errors

**Rule:** When AI produces a wrong or misleading output, capture the lesson.

**How to capture it:**
1. **In the PR / review:** Comment on what the AI got wrong and why
2. **In the project CLAUDE.md or equivalent:** Add a note so future sessions start with this knowledge
3. **In the FORGE knowledge pack:** If it's a recurring AI error in a specific domain, update the relevant knowledge pack
4. **In the team retrospective:** Share AI failure modes so the team learns collectively

**Common AI errors in software projects:**
- Hallucinated API methods that don't exist in the specified library version
- COBOL analysis that misreads paragraph logic flow
- Architecture diagrams that miss implicit dependencies
- Security review that misses context-specific vulnerabilities
- Test cases that test the happy path but miss critical edge cases

---

## 8. Model Governance: Track What Was Used

**Rule:** Record which AI model/version produced significant artifacts.

**Why:** AI models are updated. A finding from model version X may not be reproducible with version Y. Audit trails need to know what was used.

**Minimum tracking:**
- Tool name (e.g., Claude Sonnet 4.6, GitHub Copilot GPT-4o, watsonx.ai granite-34b)
- Date (model versions change over time)
- The output artifact reference

**This information goes in:**
- File headers of generated artifacts
- The Coordinator Agent delivery package audit trail
- Release notes when AI-generated components go to production

---

## Summary: The AI Usage Principles

1. **Transparent** — label AI work so reviewers know what they're reviewing
2. **Human-authoritative** — AI advises; humans decide on anything consequential
3. **Bias-aware** — verify AI recommendations fit the real context
4. **Data-responsible** — no production PII, credentials, or sensitive data in prompts
5. **IP-clear** — use enterprise-licensed tools; understand ownership
6. **Review-equal** — AI output gets the same (or higher) review rigor as human output
7. **Learning-oriented** — capture AI errors so the team improves its use over time
8. **Model-tracked** — record what was used so outputs are auditable
